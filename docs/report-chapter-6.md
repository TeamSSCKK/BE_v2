## 6-2. 핵심 기능 구현 범위

본 프로젝트에서 담당한 백엔드 구현 범위는 참가자의 출발 위치와 음식 취향 정보를 저장하고, 이를 바탕으로 모임 장소 및 식당 후보를 추천하기 위한 API와 데이터 구조를 구축하는 것이다. 모임 생성·참여·투표·최종 결정 흐름은 팀원이 담당하며, 본 구현에서는 해당 기능과 연동할 수 있도록 추천 결과를 후보 테이블에 저장하고 모임 상태를 변경하는 부분까지 포함하였다.

현재 구현한 핵심 기능은 다음과 같다.

1. Supabase PostgreSQL 기반 추천 데이터 모델 구축
   - 참가자 위치 및 음식 취향 저장
   - 장소 후보, 참가자별 이동시간, 식당 후보 저장
   - 추천 점수, 추천 순위 및 계산 방식 저장
   - 투표 및 최종 결정 테이블과의 외래키 연결

2. 서울 지역 입력 검증
   - 참가자 출발지와 추천 후보의 위도·경도를 서울 범위로 제한
   - 잘못된 좌표가 DB에 저장되지 않도록 CHECK 제약조건 적용

3. 서울 지하철역 후보 데이터 구축
   - 서울 열린데이터광장의 `subwayStationMaster` API 연동
   - 전체 원본 784건 중 서울 좌표 범위에 해당하는 505건을 DB에 동기화
   - 노선별 중복 역은 저장하되 추천 단계에서 역명 기준으로 중복 제거

4. 모임 장소 추천 API 구현
   - 모임 ID와 초대 코드를 이용한 요청 검증
   - 참가자 출발 위치 조회
   - 참가자 중심점 주변의 지하철역 최대 30개를 1차 후보로 선정
   - 이동시간과 공정성 점수를 계산하여 상위 5개 반환
   - 추천 장소와 참가자별 이동시간을 DB에 저장
   - 추천 완료 후 모임 상태를 `PLACE_VOTING`으로 변경

5. 운영 및 보안 기능
   - Supabase Edge Functions를 이용한 서버리스 API 구현
   - RLS 활성화와 `service_role` 기반 서버 작업 분리
   - 외부 API 키를 Supabase Secrets와 `.env.local`로 관리
   - 서울 지하철역 동기화 API에 별도의 관리자 토큰 적용

현재 장소 이동시간은 거리 기반 추정 방식으로 구현되어 있다. ODsay API를 이용한 실제 대중교통 이동시간과 네이버 검색 API를 이용한 식당 추천은 다음 구현 단계에서 기존 추천 인터페이스에 연결할 예정이다.

## 6-3. 추천 알고리즘 설계

### 6-3-1. 장소 추천 절차

장소 추천 알고리즘은 모든 참가자에게 이동 부담이 지나치게 집중되지 않도록 평균 이동시간뿐 아니라 최대 이동시간과 참가자 간 이동시간 편차를 함께 고려한다.

처리 절차는 다음과 같다.

1. 모임에 참여한 사용자들의 출발 위치를 조회한다.
2. 모든 출발 위치의 위도와 경도 평균을 계산하여 중심점을 구한다.
3. 서울 지하철역 데이터 중 중심점에서 가까운 역을 최대 30개 선택한다.
4. 환승역처럼 동일한 이름을 가진 역은 하나의 후보로 통합한다.
5. 각 참가자의 출발지에서 각 후보역까지 예상 이동시간을 계산한다.
6. 후보별 평균 이동시간, 최대 이동시간, 표준편차를 계산한다.
7. 공정성 비용이 낮은 순서로 정렬하여 상위 5개를 추천한다.

현재 거리 기반 예상 이동시간은 두 좌표 사이의 대권거리를 계산하는 Haversine 공식을 사용한다. 실제 이동 경로는 직선거리보다 길기 때문에 1.28의 우회 계수를 적용하고, 교통수단별 평균 속도와 고정 대기시간을 반영하였다.

```text
경로 추정 거리 = 직선거리 × 1.28

예상 이동시간 = (경로 추정 거리 ÷ 평균 속도) × 60 + 고정시간
```

대중교통은 평균 시속 21km와 고정 대기시간 10분을 사용하며, 자동차는 평균 시속 26km와 고정시간 6분을 사용한다. 이는 실제 교통 API 연동 전 알고리즘과 전체 흐름을 검증하기 위한 MVP 추정식이다.

### 6-3-2. 공정성 점수

각 후보 장소의 공정성 비용은 다음 식으로 계산한다.

```text
공정성 비용 = 평균 이동시간 × 0.50
            + 최대 이동시간 × 0.25
            + 이동시간 표준편차 × 0.25
```

- 평균 이동시간: 전체 참가자의 총 이동 부담을 줄이는 지표
- 최대 이동시간: 특정 참가자에게 과도한 이동이 집중되는 것을 방지하는 지표
- 표준편차: 참가자 사이의 이동시간 불균형을 줄이는 지표

점수가 낮을수록 전체 이동 부담과 참가자 간 편차가 작은 장소이다. 단순한 지리적 중심점 대신 실제 후보역별 점수를 비교하므로 사용자가 만나기 쉬운 교통 거점을 추천할 수 있다.

### 6-3-3. 외부 API 확장 설계

추천 계산 로직과 이동시간 산출 로직을 분리하여 추후 ODsay API 결과로 교체할 수 있도록 설계하였다. ODsay 연동 후에는 참가자와 후보역 조합별 실제 대중교통 시간을 가져오고, 기존 공정성 점수 계산식은 그대로 사용한다. 외부 API 장애나 호출 한도 초과 시에는 현재의 거리 기반 계산을 fallback으로 사용할 수 있다.

식당 추천은 최종 장소 주변의 음식점을 네이버 검색 API로 조회한 후 선호 음식, 비선호 음식, 알레르기 및 가격대를 점수화하는 방식으로 확장할 예정이다.

## 6-4. 상세 설계 및 설계 패턴 적용

### 6-4-1. 계층 분리

백엔드는 Supabase PostgreSQL과 Edge Functions를 중심으로 구성하였다.

- 데이터 계층: `meeting`, `participant`, `participant_location`, `participant_preference`, `seoul_transit_hub`, `place_candidate`, `place_candidate_travel` 등의 테이블
- 도메인 로직 계층: 거리 계산, 이동시간 추정, 후보 선별, 공정성 점수 계산
- API 계층: 요청 검증, DB 조회, 추천 실행, 결과 저장, HTTP 응답 처리
- 외부 연동 계층: 서울 열린데이터광장 동기화 및 향후 ODsay·네이버 API 연동

추천 점수 계산은 `scoring.ts`의 순수 함수로 분리하였다. 따라서 DB나 HTTP 요청 없이도 단위 테스트가 가능하며, 외부 API 연동 이후에도 계산 로직을 재사용할 수 있다.

### 6-4-2. 적용한 설계 패턴

1. Adapter 패턴

설계 다이어그램의 `TransitApi`, `MapApi` 인터페이스와 `ODsayAdapter`, `NaverMapAdapter` 구조를 반영하였다. 현재 구현에서는 이동시간 계산을 거리 기반 함수가 담당하지만, 이후 ODsay Adapter가 같은 형태의 이동시간 값을 반환하도록 교체할 수 있다. 네이버 검색 API 역시 외부 응답을 내부 식당 후보 모델로 변환하는 Adapter로 확장한다.

2. Strategy 패턴

이동시간 계산 방식을 `DISTANCE_FALLBACK`과 `ODSAY`로 구분하여 저장한다. 추천 로직은 어떤 계산 전략을 사용했는지와 무관하게 동일한 `minutes` 값을 이용하여 점수를 계산한다. 이를 통해 개발 환경에서는 거리 기반 전략을 사용하고, 운영 환경에서는 ODsay 전략을 선택할 수 있다.

3. Repository 역할 분리

Edge Function에서 Supabase 클라이언트를 통해 모임, 참가자, 후보 데이터를 조회하고 저장한다. 추천 계산 함수는 DB 접근을 직접 수행하지 않으므로 데이터 접근 책임과 계산 책임이 분리된다.

4. DTO 및 응답 모델

프론트엔드가 기대하는 `places`와 `origins` 구조로 API 응답을 구성하였다. 내부 DB 컬럼명을 그대로 노출하지 않고 `memberId`, `memberName`, `averageMinutes`와 같은 명확한 API 필드로 변환한다.

5. Fallback 패턴

외부 교통 API가 아직 연결되지 않았거나 장애가 발생한 경우 거리 기반 계산을 사용할 수 있도록 하였다. `calculation_method` 컬럼을 통해 어떤 방식으로 계산된 결과인지 추적할 수 있다.

### 6-4-3. 데이터 무결성과 보안

- 모임 상태, 참가자 역할, 교통수단, 투표 유형을 CHECK 제약조건으로 제한
- 동일 참가자의 중복 투표를 부분 UNIQUE INDEX로 방지
- 모임 삭제 시 참가자와 추천 후보가 함께 삭제되도록 외래키와 CASCADE 적용
- 익명 참가자를 식별하기 위한 UUID 접근 토큰 추가
- RLS를 활성화하고 서버 내부 작업은 `service_role`로 실행
- API 키는 Git에 포함하지 않고 Supabase Secrets로 관리
- 지하철역 동기화 함수는 `x-sync-token` 관리자 헤더로 보호

## 6-5. 주요 구현 결과

### 6-5-1. DB 및 API 배포

Supabase 프로젝트에 총 4개의 마이그레이션을 적용하였다. 기존 수동 생성 스키마를 유지하면서 추천 기능에 필요한 컬럼과 테이블을 점진적으로 추가하였다. `health`, `recommend-places`, `sync-seoul-stations` Edge Function을 배포하였으며 실제 HTTP 요청으로 정상 응답을 확인하였다.

### 6-5-2. 서울 지하철역 데이터 구축

서울 열린데이터광장 API의 역사 마스터 정보 784건을 조회하였다. 이 중 프로젝트의 서비스 범위인 서울 좌표 조건을 만족하는 505건을 `seoul_transit_hub` 테이블에 동기화하였다. 기존 초기 seed와 함께 활성 데이터는 509건이며, 추천 시 동일 역명의 노선별 중복을 제거한다.

### 6-5-3. 장소 추천 통합 테스트

서울 서부, 남부, 북부에 위치한 3명의 임시 참가자 데이터를 생성하여 실제 배포 환경에서 통합 테스트를 수행하였다.

- 입력 위치: 김포공항 인근, 강남역 인근, 노원역 인근
- 1차 후보: 참가자 중심점 주변 지하철역 30개
- 최종 반환: 상위 후보 5개
- 후보별 참가자 이동시간 저장 확인
- 추천 완료 후 모임 상태가 `PLACE_VOTING`으로 변경되는 것을 확인
- 테스트 종료 후 임시 모임과 관련 데이터를 모두 삭제

전체 역사 데이터를 사용한 테스트에서는 홍제역, 무악재역, 독립문역, 서대문역, 충정로역 등이 상위 후보로 반환되었다. 이는 서울 전체 역 데이터 중 중심점 주변 후보를 동적으로 탐색하고 있음을 보여준다.

### 6-5-4. 자동 검증

GitHub Actions에 Deno 타입 검사와 추천 알고리즘 단위 테스트를 구성하였다. 다음 항목을 자동으로 검증한다.

- Edge Function TypeScript 타입 오류
- 동일 좌표 거리 계산 결과
- 가까운 후보가 우선 추천되는지 여부
- 추천 후보 수 제한
- 환승역 중복 제거와 중심점 인근 후보 선택

모든 구현 커밋에서 CI가 성공하는 것을 확인하였다.

## 6-6. 트러블 슈팅

### 6-6-1. Supabase 프로젝트 URL 불일치

초기 자료에서 대시보드 프로젝트 ID와 REST API 프로젝트 ID가 서로 달랐다. 잘못된 프로젝트에 스키마나 함수를 배포할 위험이 있었기 때문에 프로젝트 URL을 `oufchidafmrxgympbcqo`로 통일하고, Supabase CLI 연결 시 프로젝트 목록과 권한을 확인하였다.

### 6-6-2. anon key와 service role key 혼동

초기 `.env.local`의 `SUPABASE_SERVICE_ROLE_KEY`에 anon key가 입력되어 DB 쓰기 요청이 `401 Unauthorized`로 실패하였다. JWT payload의 `role`과 프로젝트 `ref`만 확인하여 키 값 자체를 노출하지 않고 문제를 진단하였다. 이후 실제 `service_role` 키로 교체하여 서버 쓰기 작업을 정상화하였다.

### 6-6-3. 기존 DB 컬럼과 신규 추천 지표의 타입 충돌

기존 DB에 `max_travel_time`이라는 `time with time zone` 타입 컬럼이 존재하였다. 추천 알고리즘에서 계산한 최대 이동시간 정수를 해당 컬럼에 저장하면서 `invalid input syntax for type time with time zone` 오류가 발생하였다.

기존 컬럼을 변경하면 다른 팀원의 기능에 영향을 줄 수 있으므로 다음과 같이 새로운 컬럼을 추가하였다.

- `max_travel_minutes`
- `travel_time_stddev_minutes`
- `recommendation_score`

이를 통해 기존 스키마와의 호환성을 유지하면서 추천 지표의 단위도 명확하게 표현하였다.

### 6-6-4. 수동 적용 DB와 마이그레이션 기록 불일치

초기 SQL은 Supabase SQL Editor에서 수동으로 적용되어 있었지만 CLI의 마이그레이션 기록에는 존재하지 않았다. 그대로 `db push`를 수행하면 이미 존재하는 테이블을 다시 생성할 수 있었다. 따라서 기존 001·002 마이그레이션을 `applied` 상태로 복구한 후 신규 003·004 마이그레이션만 적용하였다.

### 6-6-5. 외부 API 키와 배포 환경 관리

네이버 Client Secret, ODsay 키, 서울 열린데이터광장 키를 프론트 코드나 Git 저장소에 포함하면 키가 노출될 수 있다. 로컬에서는 `.env.local`, 배포 환경에서는 Supabase Secrets를 사용하도록 분리하였다. `.env.local`과 로컬 CLI 도구는 `.gitignore`에 등록하였다.

### 6-6-6. ODsay 서버 키의 고정 IP 문제

ODsay 서버 플랫폼 키는 호출 서버의 공인 IP 등록이 필요하지만 Supabase Edge Functions는 고정 출발 IP를 보장하지 않는다. 따라서 현재 단계에서는 거리 기반 계산을 fallback으로 구현하여 전체 추천 흐름을 우선 완성하였다. 이후 URI 키를 이용한 프론트 호출 또는 고정 IP 서버를 통한 프록시 구성을 검토할 예정이다.

### 6-6-7. 테스트 요청의 한글 인코딩 문제

PowerShell을 이용한 REST 통합 테스트 중 한글 참가자 이름이 `??`로 변환되는 현상이 발생하였다. 이는 서비스 API의 문제가 아니라 테스트 셸의 문자 인코딩 문제였다. 테스트 데이터는 ASCII 이름을 사용하도록 변경하고, 실제 프론트 요청은 UTF-8 JSON으로 전송하도록 구성하였다.
